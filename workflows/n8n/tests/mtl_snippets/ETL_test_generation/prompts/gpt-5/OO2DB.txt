You are generating **draft semantic test cases** for an ETL transformation task.

## Scope and outputs
Produce **only**:
1. `semantic_cases.json`
2. One or more **source model files** referenced by the JSON (valid instances of the provided **source metamodel**)

Do **not** produce Java, JUnit, or any executable test harness code.

## Primary goal
Create semantic test cases that validate transformation behavior at the model level using task-generic assertions. Cover both typical and edge scenarios (including empty/minimal structures, recursive/self-referential structures where applicable, multiplicities, containment/opposites, and invalid-or-unmapped-risk scenarios when meaningful).

## Required JSON schema style
Use a task-generic case structure with assertions that use only these canonical payloads:

- `count`  
  ```json
  {"kind":"count","model":"<target|aux>","type":"<QualifiedTypeOrType>","expected":<int>}
  ```

- `featureValues`  
  ```json
  {"kind":"featureValues","model":"<target|aux>","type":"<Type>","feature":"<featureName>","expected":["<string>"]}
  ```

- `pathValues`  
  ```json
  {"kind":"pathValues","model":"<target|aux>","type":"<Type>","path":"<nav.path>","expected":["<string>"]}
  ```

- `objects`  
  ```json
  {"kind":"objects","model":"<target|aux>","type":"<Type>","features":["f1","f2"],"expected":[{"f1":"...","f2":"..."}]}
  ```

- `referencePairs`  
  ```json
  {"kind":"referencePairs","model":"<target|aux>","type":"<Type>","source":"<sourceRef>","target":"<targetRef>","expected":[{"source":"...","target":"..."}]}
  ```

## Hard constraints
- Do **not** use transformation-specific custom assertion fields.
- Do **not** use alternate assertion keys such as:  
  `value`, `values`, `equals`, `ids`, `pairs`, `sourceType`, `targetType`, `where`, `match`.
- Keep assertions purely semantic and deterministic.
- Ensure all expected values are explicit and concrete.
- Ensure source models are well-formed against the provided source metamodel.

## Input artifacts to use
- Reference ETL transformation file and code
- Source and target metamodels
- ETL grammar excerpt (for contextual understanding)

## Test design guidance
- Derive assertions from observable target/auxiliary model semantics implied by rules, guards, pre/post blocks, and helper operations.
- Include coverage for:
  - rule triggering and non-triggering conditions
  - guard-dependent behavior
  - helper/operation-derived naming or typing effects
  - trace/auxiliary model effects if produced by the transformation
- Prefer multiple small, focused cases over one monolithic case.
- Make case names descriptive and stable.

## Output format expectation
Return:
- `semantic_cases.json` content
- Then each generated source model file content (with filename), consistent with case references in JSON.
